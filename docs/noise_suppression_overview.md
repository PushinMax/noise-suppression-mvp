# Обзор проекта по шумоподавлению речи

См. также:

- [noise_suppression_mvp.md](./noise_suppression_mvp.md)
- [noise_suppression_dataset_stack.md](./noise_suppression_dataset_stack.md)

## 1. Что именно мы хотим построить

Этот проект **не** про «выделить русскую речь из смеси нескольких говорящих» и не про общую задачу source separation.

Рабочая постановка такая:

- вход: зашумленная одноканальная аудиозапись, где целевая речь — **русская речь**
- выход: та же запись, но с **подавленным фоновым шумом**
- цель: сохранить разборчивость и естественность речи, одновременно уменьшая неречевые фоновые помехи

Иными словами, мы хотим построить модель **speech enhancement / denoising** для русской речи в реальных бытовых условиях:

- офис
- кафе
- улица
- транспорт
- микрофон ноутбука / телефона
- вентилятор / кондиционер / клавиатура / комнатный шум

Модель должна подавлять:

- стационарный шум: вентилятор, гул, кондиционер
- нестационарный шум: посуда, трафик, клавиатура, толпа, щелчки
- умеренную реверберацию помещения, если это возможно

На первом этапе модель **не** должна решать:

- перекрывающуюся речь нескольких активных говорящих
- диаризацию
- wake-word detection
- разделение музыки и речи
- студийную реставрацию широкополосного аудио

## 2. Почему такая постановка выглядит перспективно

Этот домен интересен тем, что дает реалистичный шанс обыграть generic open-source baseline-ы **на узком домене**, даже если мы не обыграем их на всех публичных бенчмарках.

Почему это выглядит правдоподобно:

- есть достаточно открытых данных с чистой русской речью
- есть крупные открытые библиотеки шумов
- синтетическая генерация noisy-clean пар — стандартный и относительно дешевый путь
- оценивать можно не только по сигналу, но и по **WER русского ASR**
- многие сильные baseline-модели обучались на широких multilingual или англоязычных данных, поэтому domain adaptation может дать выигрыш

## 3. Кандидаты на стек данных

### Чистая речь

- **Golos / OpenSLR 114**
  - около 1240 часов русской речи
  - подходит как основной источник чистой речи
  - источник: <https://www.openslr.org/114/>

- **DNS Challenge clean speech pools**
  - multilingual clean speech, включая русские подмножества в поздних релизах
  - полезно для совместимости с baseline-рецептами и дополнительного разнообразия
  - источник: <https://github.com/microsoft/DNS-Challenge>

### Шумы

- **MUSAN**
  - классический открытый корпус с категориями noise, music и speech
  - хорошо подходит для широкого синтетического обучения
  - источник: <https://www.openslr.org/17/>

- **FSD50K**
  - 51k размеченных звуковых клипов с большим количеством реальных sound events
  - полезен для сборки доменно-ориентированной библиотеки шумов: кафе, клавиатура, транспорт, бытовые звуки
  - источник: <https://zenodo.org/records/4060432>

- **DNS Challenge noise и RIR assets**
  - полезны для стандартных speech enhancement-рецептов
  - источник: <https://github.com/microsoft/DNS-Challenge>

### Внешние ориентиры по реалистичности

- **CHiME**
  - экосистема соревнований по работе с реальной шумной речью, полезна как ориентир по реалистичности задачи
  - источник: <https://www.chimechallenge.org/>

- **URGENT Challenge 2024**
  - полезное напоминание, что хорошие результаты на простых benchmark-ах не гарантируют устойчивости в реальных условиях
  - источник: <https://urgent-challenge.github.io/urgent2024old/>

## 4. Предлагаемая схема оценки

Не стоит сводить оценку модели к одному числу.

Основные метрики проекта:

- **WER** русского ASR до и после шумоподавления
- **DNSMOS** для перцептивного качества
- **STOI** для разборчивости речи
- **SI-SDR** или близкая к SDR объективная метрика сигнала

Дополнительные метрики:

- задержка
- real-time factor
- число параметров
- потребление памяти

Для MVP наиболее убедительный сценарий выигрыша выглядит так:

> более низкий WER русского ASR на нашем целевом шумном домене по сравнению с сильным generic open-source baseline-ом при сопоставимых перцептивных метриках.

## 5. Базовые модели, на которые стоит ориентироваться

Ниже перечислены первые baseline-семейства, которые стоит явно зафиксировать.

### DeepFilterNet

- тип: causal real-time speech enhancement
- сильная сторона: практичность и низкая задержка
- почему важен: очень сильный открытый baseline для видеозвонков и live noise suppression
- официальный репозиторий: <https://github.com/Rikorose/DeepFilterNet>
- статья: <https://www.isca-archive.org/interspeech_2023/schroter23b_interspeech.pdf>

Заявленные результаты на VoiceBank+DEMAND из demo-статьи Interspeech 2023:

- **DeepFilterNet**: PESQ `2.81`, CSIG `4.14`, CBAK `3.31`, COVL `3.46`, STOI `0.942`
- **DeepFilterNet2**: PESQ `3.08`, CSIG `4.30`, CBAK `3.40`, COVL `3.70`, STOI `0.943`
- **DeepFilterNet3**: PESQ `3.17`, CSIG `4.34`, CBAK `3.61`, COVL `3.77`, STOI `0.944`

Дополнительно в той же статье указано:

- single-thread real-time factor: `0.19` на `i5-8250U`
- частота дискретизации: `48 kHz`
- общая задержка: `40 ms`

Интерпретация:

- это самый сильный **практический** baseline для нашей первой итерации
- даже если другая модель дает более высокий PESQ, DeepFilterNet сложно обыграть по deployability

### FullSubNet

- тип: spectrogram-based full-band + sub-band модель
- сильная сторона: сильный baseline для single-channel enhancement с большим следом в research-литературе
- официальный репозиторий: <https://github.com/Audio-WestlakeU/FullSubNet>
- замечание: разумно использовать как сильный research baseline, особенно для offline или менее latency-constrained сценариев

Почему его стоит держать в списке:

- это частая точка отсчета в speech enhancement-статьях
- это хорошая альтернатива, если нам нужен более research-oriented baseline, чем DeepFilterNet

### MP-SENet

- тип: сеть с параллельной оценкой magnitude и phase
- сильная сторона: сильное качество enhancement на стандартных benchmark-ах
- официальный репозиторий: <https://github.com/yxlu-0102/MP-SENet>
- статья: <https://arxiv.org/abs/2308.08926>

Почему его стоит держать в списке:

- полезен как более сильная quality-oriented точка отсчета
- вероятно, менее привлекателен, чем DeepFilterNet, для строгого real-time deployment, но полезен для оценки запаса по качеству

### ZipEnhancer

- тип: dual-path Zipformer-based модель для monaural speech enhancement
- сильная сторона: очень сильные недавние benchmark-метрики при умеренном размере модели
- статья: <https://arxiv.org/abs/2501.05183>

В abstract статьи заявлено:

- PESQ `3.69` на **DNS 2020 Challenge**
- PESQ `3.63` на **VoiceBank+DEMAND**
- `2.04M` параметров
- `62.41G` FLOPS

Интерпретация:

- это уже скорее ориентир на recent SOTA-quality, чем на легкий практический baseline
- обыграть такую модель “в среднем по всем датасетам” за 60-100 часов нереалистично
- но обыграть generic baseline-ы на **узком домене русской шумной речи** все еще выглядит реалистично

## 6. Рекомендация по стартовой архитектуре

На первом этапе проекта лучшая стратегия — **не** придумывать принципиально новую архитектуру.

Рекомендуемый путь:

1. использовать **DeepFilterNet** как основной deployment-oriented baseline
2. построить synthetic training pipeline под русский домен
3. fine-tune или адаптировать causal STFT-based enhancement модель
4. оценивать не только objective-метрики, но и русский ASR WER

Наиболее реалистичный первый выбор архитектуры:

- **causal STFT-based denoiser**
- цель: complex mask или magnitude+phase refinement
- частота дискретизации: начать с `16 kHz` для MVP
- позже, если потребуется, перейти к `48 kHz`

Почему не стоит начинать с diffusion-модели или полностью новой generative-архитектуры:

- слишком высокий инженерный риск для заданного бюджета времени
- заметно сложнее inference
- сложнее добиться real-time работы
- ниже шанс быстро получить измеримый выигрыш

## 7. Рабочая исследовательская гипотеза

Гипотеза, которую мы хотим проверить:

> Domain-adapted модель шумоподавления для русской речи, обученная на целевых synthetic mixtures и дополненная небольшим real evaluation set, может обыграть generic open-source denoiser-ы на русской шумной речи, особенно по downstream ASR WER.

Это значительно реалистичнее, чем цели уровня:

- «обыграть все open-source speech enhancement модели везде»
- «сделать лучший denoiser в мире»

## 8. Рекомендуемый следующий шаг

Следующий конкретный документ должен зафиксировать **MVP dataset stack**:

- точные источники чистой речи
- точные подмножества шумов
- рецепт смешивания
- разбиение на train / val / test
- план сбора real-world evaluation set
- план запуска baseline-моделей

Когда это будет зафиксировано, можно окончательно выбрать домен и начинать реализацию.
